# create-docs

`docs/` 안에 프로젝트 문서를 **페르소나 → 유저 시나리오 → 아키텍처 → DB → API** 순으로 만든다.
각 문서는 앞 문서를 참조해 일관성을 유지한다.

---

## 시작하기 전에

1. `docs/` 폴더의 기존 파일을 확인해 어디까지 작성되어 있는지 파악한다.
2. 이미 작성된 문서는 건너뛰거나 업데이트 여부를 사용자에게 묻는다.
3. 없는 파일부터 순서대로 진행한다.

---

## 공통 원칙

- **인터뷰 → 초안 → 확인 → 저장** 4단계를 각 문서마다 반복한다.
- 초안을 작성하기 전에 반드시 핵심 질문에 대한 답을 얻어라. 질문은 한꺼번에 묶어서 한 번에 물어본다.
- 초안은 대화 안에서 먼저 보여주고, 사용자가 확인한 뒤 파일에 쓴다.
- 다음 문서를 작성할 때는 이전에 확정한 내용을 그대로 가져다 쓴다. 다시 묻지 않는다.

---

## 단계 1 — 페르소나 (`docs/personas.md`)

### 질문할 내용
- 앱 이름과 한 줄 목적은?
- 주요 사용자 유형은 몇 가지이며, 각각의 역할과 기술 수준은?
- 각 유형이 앱에서 가장 자주 하는 일 3가지는?

### 출력 포맷
```markdown
# 페르소나 — [앱 이름] (MVP)

> [MVP 헌장 또는 범위 한 줄 요약]

| 역할 | 기술 수준 | 주요 시나리오 |
|------|-----------|--------------|
| **[역할명] ([이름], [나이])** [역할 설명] | [초급·중급·고급] — [익숙한 도구] | ① ... ② ... ③ ... |
```

### 저장 위치
`docs/personas.md`

## 단계 2 — 유저 시나리오 (`docs/user-stories.md`)

> 페르소나를 반드시 먼저 읽고 시작한다.

### 질문할 내용
- 어느 페르소나의 하루 흐름을 핵심 시나리오로 선택할 것인가?
- 해당 페르소나의 업무 시작·종료 시각과 주요 행동 포인트는?
- 현재 결정되지 않은 정책(권한, 다중 담당자, WIP 제한 등)이 있는가?

### 출력 포맷
```markdown
# User Stories — [앱 이름]

## 1. 핵심 시나리오

**페르소나:** [이름] ([역할]) — [한 줄 설명]

---

### 하루 흐름 (시간 순)

| 시각 | 행동 | 앱에서 하는 일 |
|------|------|----------------|
| **HH:MM** [상황] | [행동 요약] | [UI 상호작용] |

---

### 핵심 기능 매핑

| 시나리오 단계 | 사용 기능 |
|--------------|-----------|
| ... | ... |

---

## 2. 결정 사항

1. **[미결 정책 질문]?** → [현재 결정 또는 '추후 결정 필요']
```

### 저장 위치
`docs/user-stories.md`

## 단계 3 — 아키텍처 (`docs/architecture.md`)

> 유저 시나리오의 기능 목록을 보고 기술 스택을 선택한다.

### 질문할 내용
- 프론트엔드 프레임워크는? (기본: Next.js)
- 백엔드·DB는? (기본: Supabase/Postgres)
- 배포 환경은? (기본: Vercel)
- 인증 방식은? (기본: Google OAuth)
- 추가로 포함할 외부 서비스가 있는가?

### 출력 포맷
```markdown
# 아키텍처 — [앱 이름] (MVP)

```
  [Browser]
      │  [인증 방식]
      ▼
 [프론트엔드 + 런타임]  ──API Routes──▶  [DB/백엔드]
  ([설명])                               ([설명])
```

## 기술 스택

| 레이어 | 선택 | 이유 |
|--------|------|------|
| 프론트엔드 | ... | ... |
| 런타임 | ... | ... |
| DB | ... | ... |
| 인증 | ... | ... |
| 배포 | ... | ... |
```

### 저장 위치
`docs/architecture.md`

## 단계 4 — DB (`docs/db.md`)

> 아키텍처에서 결정한 DB 기술과 유저 시나리오의 기능 목록을 바탕으로 테이블을 설계한다.

### 질문할 내용
- 핵심 테이블 이름(들)은? (MVP는 보통 1–2개)
- 각 테이블에 필요한 컬럼과 타입은?
- 상태(status) 컬럼이 있다면 허용 값은?
- 외래키 관계가 있는가?

### 출력 포맷
```markdown
# DB 설계 — [앱 이름] (MVP)

```sql
create table [테이블명] (
  id          uuid primary key default gen_random_uuid(),
  [컬럼명]   [타입] [제약],
  created_by  uuid references auth.users(id),
  created_at  timestamptz not null default now()
);
```

> MVP 제약: ENUM·인덱스·트리거·RLS 추가 금지. 상태값은 text check로만 제한.
```

### 저장 위치
`docs/db.md`

## 단계 5 — API (`docs/api.md`)

> DB 테이블을 보고 필요한 엔드포인트를 도출한다. 엔드포인트는 최대 6개로 제한한다.

### 질문할 내용
- 클라이언트가 서버에서 직접 받아야 할 데이터(필터, 목록 등)가 있는가?
- 모든 API에 인증이 필요한가, 일부 공개 엔드포인트가 있는가?
- 기본 CRUD 외에 특수한 액션(로그아웃, 일괄 처리 등)이 필요한가?

### 출력 포맷
```markdown
# API 엔드포인트 — [앱 이름] (MVP)

| METHOD | PATH | 설명 | 인증 |
|--------|------|------|------|
| GET | /api/[resource] | 목록 조회 | 필요 |
| POST | /api/[resource] | 생성 | 필요 |
| PATCH | /api/[resource]/[id] | 수정 | 필요 |
| DELETE | /api/[resource]/[id] | 삭제 | 필요 |
```

> base path: `/api` · 버전 prefix 없음 · `created_by`는 서버에서 주입

### 저장 위치
`docs/api.md`

## 완료 후

모든 문서가 작성되면 다음을 사용자에게 안내한다.

- 생성된 파일 목록과 경로
- `CLAUDE.md`의 `docs/` 항목이 최신 상태인지 확인 권고
